My Approach to Reproducible Science

Now I‚Äôll spend just a few moments outlining some of the steps I find most important to me when running open, reproducible, and shareable projects.

For code block:

My current go-to pipeline tool in R is the targets package. (Here is code for a simplified example). Targets allows for a whole project, from reading in raw data to producing a manuscript, to be controlled using a single command. These tools enforce an organized code structure ‚Äì no more thousand-line do-it-all scripts ‚Äì reduce human error, and make the flow of the project clear.

  • Communication: what are the inputs and desired outputs of a project?

(step one or even step zero: Communication) It‚Äôs critical to find out what the project needs are, especially the data and the expectations for project outcomes. Clearly defining end points is crucial to not waste time on the all the hard work that goes on between.

  • Developing a pipeline to keep things organized so colleagues can rerun, verify, and adapt results without guesswork

Developing a pipeline: A simple file structure and the use of a pipeline manager keep a project organized, straighforward to run, and easy to maintain.

  • Control the environment, e.g., using Docker and renv, to prevent the ‚Äúwell, it worked on my computer‚Äù problem

For others to easily see and repeat what you did, it‚Äôs crucial they can run the analysis using the same environment you did. That‚Äôs where Docker comes in to create a consistent computing environment regardless of the computer you‚Äôre using, and using something like renv ensures that package versions are the same as those used in development.

  • Use version control (e.g., git and GitHub) to allow others inside and outside the project to contribute and reuse code

Hosting everything on platforms like github simplifies collaboration, transparency and availability of your entire project, not just the data set you used.

  • Documentation so future researchers could find and understand the work without my direct involvement

Finally, leaving behind clear documentation allows others in the future to make use of all this work even if years have gone by.
